# Mindspan 1.0.0-beta.1 deployment notes

This release must be deployed as an in-place upgrade from the current
`v0.2.0-beta.5` production database. Do not reset, reseed, or replace production
data.

## Before changing production

1. Confirm the deployed source is exactly tag `v1.0.0-beta.1`.
2. Take protected PostgreSQL 17 schema and data backups and verify both
   checksums.
3. Record the current counts for auth users, profiles, groups, memberships,
   attempts, mastery, feedback, pack unlocks, questions, and question versions.
4. Run the migration dry run. It must contain exactly these migrations, in this
   order:
   - `20260714130000_retire_assessment.sql`
   - `20260715190000_question_feedback_wrong_topic.sql`
   - `20260715200000_release_tracking.sql`
   - `20260716100000_roger_deakins_difficulty.sql`

## Database and catalog update

1. Apply the four reviewed migrations.
2. Run `npm run db:verify`.
3. Run `npm run catalog:validate`.
4. Run `npm run catalog:load` once. From the beta.5 catalog baseline, expect:
   - 0 inserted
   - 36 updated
   - 1,024 skipped
5. Run `npm run catalog:verify`. Expect 1,100 published questions across 14
   packs, with starter packs at 100 questions and expansion packs at 50.
6. Run `npm run catalog:load` a second time. Expect 0 inserted, 0 updated, and
   1,060 skipped. This is the idempotency checkpoint.

The catalog changes represent 34 directly reviewed catalog questions plus two
companion pack-rotation changes. The Roger Deakins seed-backed question is
versioned by migration. Starting from production's beta.5 total of 1,101
question versions, the expected total after this release is 1,138.

## Preservation checkpoints

- Existing auth users, profiles, groups, memberships, attempts, mastery,
  feedback, achievements, Insight transactions, media, and pack unlocks must be
  unchanged except for normal activity during the deployment window.
- The assessment retirement migration preserves historical assessment runs,
  responses, awards, Insight, and mastery.
- Release tracking backfills existing players to `0.2.0-beta.5`, allowing them
  to receive the `1.0.0-beta.1` update notice without replaying the baseline
  release.
- Do not run `supabase/seed.sql` in production.
- Take and checksum a second protected backup after migrations and catalog load.

## Activation and smoke tests

1. Build and promote the immutable standalone artifact for the exact tag.
2. Confirm direct, reverse-proxy, and public HTTPS health checks report the
   database reachable.
3. Restart the service once and repeat the health checks.
4. Verify signed-out routes redirect to login.
5. With an existing beta account, verify login persistence, the one-time update
   notice, Updates history, play/answer submission, question feedback, group
   leaderboard labels, and preserved progress.
6. With a system administrator, verify Question Quality display and export.

## Rollback

The beta.5 application can be restored as an emergency runtime rollback, but
the database and catalog changes should remain in place because they are
forward-compatible and contain preserved user history. Prefer a forward fix.
Restore a database backup only for confirmed destructive corruption and only
after separately preserving all activity created since that backup.
